## Why these methods were preferred
1. Extension overrides over CKAN core edits
- Preferred: template/CSS changes in `ckanext-customtheme`.
- Reason: preserves upgrade path, isolates custom behavior, minimizes merge risk.
- Alternative rejected: direct edits in CKAN core templates/assets due to high maintenance cost.

2. Container entrypoint automation over manual admin fixes
- Preferred: startup scripts (`docker-entrypoint.d`) and env-driven setup.
- Reason: reproducibility and persistence across container recreation.
- Alternative rejected: one-off shell fixes inside running container because they are non-repeatable.

3. Log-driven debugging over blind restarts
- Preferred: inspect precise traceback and startup logs before changes.
- Reason: reduced recovery time and avoided introducing unrelated regressions.
- Alternative rejected: repeated restart/rebuild without diagnosis.

4. Route-level HTTP smoke tests over visual-only checks
- Preferred: `curl -I` checks for key pages.
- Reason: objective and scriptable verification of availability/state codes.
- Alternative rejected: browser-only checking which can be confounded by cache/session state.

5. Incremental CSS hardening over full rewrite
- Preferred: targeted constraints (`overflow-wrap`, `word-break`, clamping, flex min-width handling).
- Reason: fixes specific breakpoints with low regression risk.
- Alternative rejected: large CSS rewrite at late stage, which would increase instability.
